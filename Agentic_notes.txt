When a LLM need to be used in an org, then fine tuning is not the right approach as the policies/data change. So RAG is better. Its called RAG because it not only retrieves but augmenting the retrieved data and then generating it. For example, if I upload HR policy document which has only leave policy and if I ask a qn on "vacation policy" then it still gets the answer by matching with leave policy. csv file is not a good source as it contains only data with no context. RAG system should always have a citation (if it should not be displayed to end users, then it should be captured internally). The citation comes as we are storing the metadata also. 

If you want to have RAG system not to augment the output then use hyperparameters. The hyperparameters are
	1) Max output tokens: For RAG system, its essential that we control how much should be the length of the response. So set max output tokens.
	2) Temperature: If you want to have RAG system not to augment the output the set the temp to 0 - this is required when we need to get the legal details. If you want it to be creative then set the temp value as 1 (some models also allow 2 - gives answer like a poet). IF its 0.15, then it will alter few words. Do note that this also helps controls tokens as additional words are not generated
	3) Top P: What is the most likely word to be returned. The sky is... It has words blue 0.60, clear 0.20, cloudy, 0.10. If top p is given 0.8 then it will give clear blue (0.60 +0.20) and top is 0.9 cloudy clear blue (0.1+0.2+0.6). If we give top p as 0.1 then it will give only blue as it will go for the word with high probability. So setting top p with lower value reduces the number of tokens generated
	4) Frequency Penalty: a setting that discourages the model from repeating the same words too often in a generated response. For ex: The house is looking good. The house a large hall. The house has lot of furnitures. Instead write: The house is looking good. It has a large hall and lot of furnitures.  So this setting act as "repetition tax". It looks at how many times a word has appeared and penalizes its probability of being chosen again. This may need not be required in RAG
	5) Presence Penalty: Presence of the same word gets penalized. Topic stagnation. While frequency is for repetitive wording
	Max output tokens and Temperature help to control Hallucination.

	Note: There are hyperparameters set during the LLM training. They are different category.

RAG will have two different set of implementations.
	1) Ingestion: Saving source data in to vector DB [Onetime setup]. Password protected files should not be sent. Check whether the source document does not have conflicting information. Noise words should be removed (pl, thank you, "assign this ticket to A if B is not in office", header, footer)
	2) Retrieval: Retrieving data from vector db

	Ingestion Pipeline:
		1) Extract content 
		2) Chunking (Fixed size - best for raw data, log, semantic chunking - FAQs knowledge bases, Recursive - real world rag systems) chunk overlap, chunk size
		3) Embedding model
		4) Embedding
		5) Save the embeddings and original text in vector DB

Vector DB:
	In case of RDBMS, and NoSQL DB, we will be able to read the data. But in case of Vector DB, it will be unreadable as data is stored as Vector embeddings. The actual conversion of word to embedding is done by embedding models. We will have to tell embedding model the number of dimensions to use. The best one is 1536 dimension for our usecase. To get best results we typically store as sentences or paras. This DB uses semantic search but RDBMS, and NoSQL DB dont have such a faciilty. Hence use vector DB.

Popular vector DB:
	1) ChromaDB, Qdrant, Pinecone, Milvus, Weaviate, FAISS (is only a library), Pgvector (FTS) - postgreqsl. You can install pgvecctor
		FAISS is a library

Chunking:
	When we store a doc as sentences as paras then its called as Chunking. If a para gets divided in to two due to chunking (one portion in chunk1 and the second portion in chunk2), then the replies will be chopped. To avoid this have chunk overlapping. 

	Retrieval Pipeline:
		For retrieval, add vector search tool and have the agent to get the data from the vector search tool.
		1) When users ask a qn
		2) Covert the qn in to embedding via the same embedding model used in Ingestion
		3) Do a vector search
		4) Get the embedding and original text 
		5) LLM will piece together the response based on the text returned

	Whatever the model sees is its context - system prompt, messages, tools. These are what we know so far.
		context = system prompt + tools schema 
			*tools schema = tool name, its description, i
	There is one more - memory - use to refine its answer based on memory.

		name: pdf
		description: Use this skill whenever the user has to do something with pdf.
		In the command line if you type /pdf, then a given md file will be created as a prod grade pdf. 
		Each agent will have acces to these skills. Whenever the agent needs the skill will be loaded. So an agent can now do multiple skills

	The max of this context is called the context size or context window.

	If an AI app has an agent that call Jira systems, then if we need to create another project then again that agent will have to call jira system. So with each newer version, the code duplicates. To avoid this use MCP. Also the template of open aI is different from Anthropic. This again create another level of issues.
	To avoid these problems, we use MCP, which is a standard for the above problem. MCP will host the jira tool which can be accessed common fashion by two AI apps, there by reducing the code duplication.
	To access MCP, we need MCP client. 
	For ex, to access Tavilly we had created an API key. Another way is to logon to its MCP server which will have its own URL - this will also require an API key. Similarly jira MCP server is also available.

	There are 3 different ways in which MCP servers are available
	1) Third party MCP server (Tavilly, Jira, GitHub)
	2) community MCP server
	3) our own custom MCP server

	before going for our own custom MCP server, check whether there are any community MCP server in GitHub. They are same like pypy packages but there could be some unknown vulnerabilities
	If this is not available go for own custom MCP server. It can be developed in any of the commonly used languages. For ex, we can use Python. To avoid typing in raw python code, we can use Python MCP SDK. It uses FastMCP as the framework. Do note that FastMCP will have it own port in which it will run (similar to streamlit)

	There are three transport layers available in MCP - transport layers are modes in which your MCP client connects to MCP server
	1. stdio - local machine can only access
